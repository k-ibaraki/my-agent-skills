# kiro-delegate

ユーザーが **明示的に** kiro への委譲を指示した場合のみ使用する（「kiroに投げて」「kiroで実装して」など）。
一般的なコーディングタスクでは絶対に自動発動しない。

---

## 0. 事前チェック（必須）

kiro はインタラクティブに停止することがある。タスク実行中にバージョン不足やログイン要求で中断されないよう、開始前にまとめてチェックする。

### 0-1. インストール・バージョン確認

```bash
VERSION=$(kiro-cli --version 2>/dev/null | grep -oE '[0-9]+\.[0-9]+\.[0-9]+')
if [ -z "$VERSION" ]; then
  echo "NOT_INSTALLED"
else
  MAJOR=$(echo "$VERSION" | cut -d. -f1)
  if [ "$MAJOR" -lt 2 ]; then echo "OLD_VERSION:$VERSION"; else echo "OK:$VERSION"; fi
fi
```

- `NOT_INSTALLED` → 「`kiro-cli` がインストールされていません。インストール後に再実行してください。」と伝えて中断
- `OLD_VERSION:X.Y.Z` → 「kiro-cli のバージョンが古いです（現在: X.Y.Z、必要: 2.0.0 以上）。`kiro-cli update` を実行してください。」と伝えて中断
- `OK:X.Y.Z` → 次のステップへ

### 0-2. ログイン確認・自動ログイン

```bash
kiro-cli whoami 2>&1; echo "EXIT:$?"
```

exit code が 0 ならログイン済み。非0 の場合はユーザーに「ログインを開始します」と伝えてから実行する：

```bash
kiro-cli login
```

`timeout` は 120000（2分）以上を設定してブラウザ認証の完了を待つ。完了後 `kiro-cli whoami` で確認してから次へ。

**注意**: ログインセッションはタスク間で期限切れになることがある。同一会話内で複数タスクを連続委譲する場合も、各タスク前にこのチェックを再実行すること。

## 1. タスクを自己完結した形に整理する

kiro はインタラクティブに質問できないため、曖昧な指示では誤った判断をしたり途中で止まったりする。タスク説明は以下を含めること：

- **対象ファイルと場所**: `src/foo.py` の line 42 など、kiro が迷わず作業できる粒度
- **期待する変更の詳細**: 変更前後の意図を明示する（「修正して」だけでは不十分）
- **完了条件**: 「品質チェックを実行して通過すること」など kiro が自己検証できる基準

> **Before/After コードの書き方について**:
> kiro は対象コードを見つけるために内部でファイル検索ツールを使う。
> Before 例に `{`, `}`, `.`, `*`, `(`, `)` などの正規表現特殊文字が含まれると、
> 検索パターンとして解釈されツールのバリデーションエラーが発生することがある。
> **ファイルパスと行番号を明示し**、Before/After はあくまで「意図の説明」として添えること。
> （例: "`src/foo.py` の line 42 の `config = {"key": "value"}` 行を修正する"）

> **複数ファイルをまとめて委任する場合**:
> ファイルごとにセクションを分けて記述する（`### ファイル1: ...` / `### ファイル2: ...`）。
> 同じパターンの変更であっても、対象メソッドと行番号をファイルごとに列挙すること。

> **TDD でインポートが削除される場合**:
> 修正によってモジュールから特定シンボルのインポートが消える場合、
> `patch("module.Foo")` を使うテストは `AttributeError` で失敗する。
> この場合は kiro にテストアプローチの変更を許容させる（kiro は自律的に回復できる）か、
> プロンプトに「モジュールが X をインポートしていないことを構造的に検証するテストを書く」と明示しておくと安定する。

> **ブランチ作成を含むタスク**:
> kiro が `git checkout -b <branch>` を実行する時点でどのブランチにいるかは制御できない。
> 作業ブランチに未マージのコミットが残っている場合、それらがそのまま新ブランチに取り込まれてしまう。
> タスク説明のブランチ作成手順は必ず **`git checkout <base-branch> && git checkout -b <new-branch>`** の形式で記述すること。

## 2. kiro-cli を実行する

**重要**: kiro はコード変更後にプロジェクトの品質チェック（テスト・型チェック・lint など）を実行するため、変更規模が小さくても完了まで数分かかる。

### タイムアウトの目安

| 規模 | 目安 | timeout_ms |
|---|---|---|
| 小（1〜2ファイル） | 6〜10分 | 600000 |
| 中（3〜5ファイル） | 15〜20分 | 1200000 |
| 大（6ファイル以上・テスト多数） | 25〜30分 | 1800000 |

> **コア動作の変更はファイル数より大きめに見積もる**:
> SSE イベント順序・メッセージフロー・ミドルウェアの処理順など、アプリのコア動作を変更するタスクでは
> 既存テストへの波及修正が大量に発生する。変更ファイル数が「小〜中」でも **大**（1800000ms）を使うこと。

タイムアウトで途中終了した場合は、完了済み部分をプロンプトに明記して再実行する。

### 2-a. タスク前の使用量を記録する

kiro の使用量をタスク前後でユーザーに報告するため、事前に保存しておく。
Monitor を挟むとシェル変数のスコープが失われるため一時ファイルを使う：

```bash
KIRO_USAGE_FILE="/tmp/kiro-usage-before-$(uuidgen | tr '[:upper:]' '[:lower:]').txt"
echo "/usage" | kiro-cli chat --no-interactive 2>&1 | sed 's/\x1b\[[0-9;?]*[mGHJKlhABCDfr]//g' | grep -v '^$' > "$KIRO_USAGE_FILE"
```

### 2-b. タスク内容を一時ファイルに書き出す

heredoc 内の特殊文字エスケープを避けるため、タスク説明をファイルに保存する。
UUID でファイル名を一意にして同名衝突を防ぐ：

```bash
KIRO_TASK_FILE="/tmp/kiro-task-$(uuidgen | tr '[:upper:]' '[:lower:]').txt"
cat > "$KIRO_TASK_FILE" << 'TASK_EOF'
<タスク説明（複数行）>
TASK_EOF
```

### 2-c. Monitor でリアルタイム表示しながら kiro を実行する

kiro の実行には **Bash ではなく Monitor** を使う。Monitor はリアルタイムに出力を転送し、kiro が長時間停止した場合も検知できるからである。実行後はタスクファイルを削除する：

```
Monitor(
  command="kiro-cli chat --no-interactive --trust-all-tools < \"$KIRO_TASK_FILE\" 2>&1 | grep --line-buffered -E 'FAILED|failed|Error:|error:|Traceback|passed in|warnings in|✅|❌|All checks|すべてのチェック|<プロジェクトの品質チェックコマンド>'; rm -f \"$KIRO_TASK_FILE\"",
  description="kiro実行中: <タスク概要>",
  timeout_ms=1200000,   # 規模に応じて調整（上の目安参照）
  persistent=false
)
```

**通知フィルターの方針**:
- 品質チェック結果（`passed in`, `FAILED`, `✅`, `❌`）
- エラー・例外（`Error:`, `Traceback`, `failed`）
- 品質チェックコマンドの実行行（プロジェクトの自動修正・検証コマンド名をフィルターに追加する）
- ルーチンのファイル read/write 成功ログは通知不要なのでフィルタで除外する

オプションの意味：
- `--trust-all-tools`: kiro がファイル変更・コマンド実行を確認なしで行う
- `--no-interactive`: kiro はタスク完了後に自動終了する（途中で止まらない）

### 実行中の対応方針

Monitor から通知が届いても、**エラー・失敗・タイムアウト以外では返信しない**。kiro のルーチン進捗はそのまま垂れ流しにする。

介入が必要な場合のみ：
- `FAILED` / `Error` → 内容を確認して対処を検討。ただし以下は介入不要（品質チェック通過の通知を待つ）：
  - **TDD の Red フェーズ**: テストを先に書くタスクでは最初の `FAILED` は正常。kiro が実装コードを書いて Green に進むまで待つ。
  - **`Tool validation failed`**: kiro が内部でリトライ中。数回繰り返されない限りは待機。
  - **品質チェック失敗からの自律回復**: kiro は失敗時にアプローチを変更することがある（例: テスト手法の切り替え）。試行錯誤が出力に現れても問題ない。
- `Monitor timed out` → kiro プロセスが終了しているか確認し、必要なら再実行

## 3. 出力を確認する

**重要**: Monitor が `status: completed` を返すまで次のステップに進まないこと。kiro は内部で複数ステップを順番に実行する（変更 → クリーンアップ → 品質チェック）。途中の通知で差分を確認すると中間状態（重複インポートなど）が見えることがある。

コマンドの返却内容で分岐する：

**A. ファイルに自動保存された場合**: 末尾にテスト結果とサマリーが集中しているため、保存ファイルの末尾だけ読めば十分。

**B. インラインで返ってきた場合**: そのままコマンド結果の末尾を読む。

どちらの場合も確認ポイントは同じ：
- 品質チェック（テスト・型チェック・lint）の結果
- kiro のサマリーコメント（何を変更したかの箇条書き）

---

## 4. 変更内容を検証する

kiro のサマリーは「何をしようとしたか」であり「何が実際に変わったか」ではない。必ず自分で確認する：

```bash
git status --short
git log <base>..HEAD --oneline
```

- **未コミットの変更がある** → 正常（コミット指示を含めなかった場合）。`git diff` で変更内容を確認する。
- **コミットが存在する** → 要確認。コミットメッセージと差分を精査し、意図しない変更やプッシュがないか確認する。コミット数が想定より多い場合は `git show <commit>` で各コミットの内容を確認してから整理方法を判断する。

差分確認では両面を検証する：
- **意図した変更が行われているか**: 指示した変更箇所がすべて反映されているか
- **意図しない変更がないか**: 無関係なファイルに副作用が出ていないか

kiro が誤った変更をしている場合は、エージェント自身がファイルを直接修正するか、補足指示を加えて kiro を再実行する。

> **環境固有の検証について（重要）**:
> kiro がタスク内で実行する動作確認は、あくまでローカル環境での確認に過ぎない。
> **kiro に委譲したとしても、結果の品質責任はエージェント自身にある。**
> 変更内容が特定の実行環境（Docker ビルド、CI など）を前提とする場合、
> kiro の「動作確認済み」を鵜呑みにせず、エージェント自身がその環境で再確認すること。
>
> 例：Dockerfile を変更した場合 → kiro のローカルコマンドテストでは不十分。
> エージェントが `docker build .` を実行してビルドが通ることを自分で確認すること。

## 5. ユーザーに報告して後続作業を実施する

- 変更ファイルと変更内容をユーザーに伝える
- 品質チェック結果を報告する
- **コミット・プッシュなどの後続作業はエージェント自身が行う（kiro にはさせない）**。kiro は `--trust-all-tools` で確認なしに push 等を実行する可能性がある。
- タスク完了後に使用量を取得してユーザーに伝える：

```bash
KIRO_USAGE_FILE=$(ls -t /tmp/kiro-usage-before-*.txt 2>/dev/null | head -1)
echo "=== タスク前 ===" && cat "$KIRO_USAGE_FILE"
echo "=== タスク後 ===" && echo "/usage" | kiro-cli chat --no-interactive 2>&1 | sed 's/\x1b\[[0-9;?]*[mGHJKlhABCDfr]//g' | grep -v '^$'
rm -f "$KIRO_USAGE_FILE"
```

## トラブルシューティング

| 症状 | 対処 |
|---|---|
| タイムアウト | `timeout_ms` を増やし、完了済み部分をプロンプトに明記して再実行 |
| 品質チェック失敗（kiro が回復できない） | エージェント自身で残りを修正する |
| タスクが途中で止まった | 出力末尾を確認して原因を特定し、補足指示で再実行 |
| MCPサーバーの警告 | 通常は無視してよい（kiro の動作には影響しない） |
| pre-commit フックでコミット失敗 | ruff format 等のフックがファイルを自動修正した可能性。`git add -u` で再ステージしてから再コミット。`--no-verify` でフックをスキップしないこと。 |
| ログインセッション切れ | `kiro-cli login` を再実行してブラウザ認証を完了する。同一会話での複数タスク連続実行時も発生しうる。 |
