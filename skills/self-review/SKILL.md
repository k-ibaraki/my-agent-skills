# Self-Review

PRを出す前に自分の変更を5観点でレビューし、問題点のみを指摘する。問題がなければその旨を報告し、あれば修正するか選択肢を提示する。

## Step 1: 差分の取得

引数があれば、それを差分対象として使う（例: `HEAD~1`、`feature/my-branch`）。

引数がなければ、リポジトリのデフォルトブランチを自動検出する:

```bash
git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's|refs/remotes/origin/||'
```

検出できなければ `main`、それも存在しなければ `dev` を試みる。

差分を取得する:

```bash
git diff <base>...HEAD
```

差分が空（クリーンブランチ）であれば、その旨を伝えて終了する。

## Step 2: プロジェクト規約ドキュメントの特定

全ドキュメントを読むと時間とコンテキストを浪費するので、規約に関連するファイルのみを絞り込む。

1. `CLAUDE.md` と `.claude/CLAUDE.md` を読む（規約の主要な情報源）
2. Glob で `docs/` 配下のファイル一覧を取得し、ファイル名・パスから規約・ガイドライン・アーキテクチャに関連するものを特定する（例: `*guideline*`, `*convention*`, `*architecture*`, `*ubiquitous*`, `*style*`）
3. Grep で `docs/` 配下のファイルから「禁止」「規約」「ルール」「must」「should」などのキーワードを検索して追加の関連ファイルを見つける
4. 上記で特定したファイルのみを読む

## Step 3: 5観点でのレビュー

差分と規約ドキュメントを照らし合わせて問題を探す。「問題がない」項目は一切出力しない。

### 1. コード品質
- 命名が意図を明確に伝えているか（変数名・関数名・クラス名）
- 関数が単一の責務に収まっているか（50行超は要注意）
- 削除・整理すべきコードがコメントアウトで残っていないか
- 不要なデバッグコードや一時的なコードが残っていないか
- **ロジック・アルゴリズムの正確性**: パス計算・API引数・データ変換の順序が意図通りか。`None` や省略時のデフォルト引数が実際に何をするか確認する（例: `model_name=None` でデフォルト重量モデルが使われていないか）
- **リソース管理**: 一時ファイル・ディレクトリ・外部接続は `with` 文（コンテキストマネージャ）で確実にクリーンアップされているか。`try/finally + 手動クリーンアップ` より `with` 文を優先する
- **名前衝突**: 一時ファイルや共有ディレクトリで同名衝突が起きる可能性はないか（UUID等で一意性を保証しているか）

### 2. テスト
- 変更した機能に対応するテストが書かれているか
- 追加したコードパスに対してエッジケースがカバーされているか
- プロジェクトのテスト規約（モックの使い方、テストの独立性など）に従っているか

### 3. セキュリティ
- ユーザー入力の検証漏れはないか（システム境界のみで検証する）
- 認証・認可のチェック漏れはないか
- シークレットや認証情報がコードにハードコードされていないか
- SQLインジェクション・XSS・コマンドインジェクションなどの脆弱性はないか

### 4. ドキュメント
- 新しいコンセプト・コンポーネント・重要な設計変更をした場合、関連ドキュメントを更新したか
- APIエンドポイントを追加・変更した場合、OpenAPI仕様の更新が必要ではないか
- 重要なアーキテクチャ決定をした場合、ADRを作成すべきではないか

### 5. プロジェクト規約
- Step 2で読んだ規約ドキュメントに違反していないか
- インポートの配置（関数内インポートがある場合、それは規約違反か）
- エラーハンドリングのパターン（プロジェクト固有のものがある場合）
- インライン型チェック抑制コメント（`# ty: ignore` 等）の禁止
- MCP固有コードやサーバー名のハードコードなど、プロジェクト固有のアンチパターン
- **SSEストリーミングの終了規約**: すべてのエラー終了パスで `end` イベントを送信してからストリームを終了しているか（`error` イベントの直後に `end` イベントが必要。`docs/api/streaming-response.md` 参照）
- **プロトコル・API契約の遵守**: 既存の他エラーパスと同じ終了シーケンスを踏んでいるか。新しいエラーパスを追加したとき、既存パスの慣習（ログ→イベント→終了の順序）と一致しているか確認する

## Step 4: 結果の出力

問題が見つかった場合のみ、以下の形式で出力する:

```
**[番号] [観点] — [問題の概要]**
場所: `ファイル名:行番号`（特定できる場合）
問題: 何が問題か
提案: どう直すか（具体的なコード例があれば示す）
```

問題がなければ「レビュー完了: 問題は見つかりませんでした」と伝えて終了する。

## Step 5: 修正の確認

問題が1件以上あった場合、`AskUserQuestion` ツールで一度だけ聞く:

> 修正する問題の番号を教えてください（例: 1,3,5 / all / none）

指定された番号の問題を修正する（Bash・Edit ツールはユーザー確認が求められる）。

`none` またはスキップ指示があればそのまま終了する。
